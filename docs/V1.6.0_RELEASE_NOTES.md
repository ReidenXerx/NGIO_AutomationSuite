## 🆕 New Features

### 🏔️ LOD Grass Cache Generation (DynDOLOD-Ready)

The tool now produces a separate **"Grass Cache - Default"** archive built specifically for DynDOLOD / TexGen LOD generation. After your seasonal cache builds, the suite optionally re-emits the `.cgid` files **without seasonal suffixes**, packaged as a ready-to-install mod.

**What you get:**

- A drop-in `Grass Cache - Default.zip` containing un-suffixed cgid files
- Direct compatibility with the DynDOLOD grass LOD pipeline
- TexGen-ready output that other LOD tools can ingest

**How to use:**

- Interactive mode prompts you about LOD grass during configuration
- The tool asks which season's cache to use as the LOD source (defaults to your first-generated season)
- All seasonal archives still get produced normally — LOD output is in addition

**Important scope note:** This release **produces the artifact** that you feed into DynDOLOD or TexGen. It does **NOT** call those tools for you — their workflows are too varied (xLODGen vs DynDOLOD vs TexGen, different load orders, etc.). You take the output and run it through your existing LOD pipeline.

### 🪟 `--no-notifications` Flag

```bash
ngio-automation --no-notifications
```

Disables all toast notifications and system sounds. Use this for:

- Wine / Proton (Steam Deck)
- Headless servers / scheduled tasks
- Overnight runs where you don't want toasts firing

The tool also auto-disables toasts when it detects Wine/Proton via `WINELOADER` / `WINEPREFIX` environment variables. Sounds keep working since they use stdlib `winsound`.

## 🐛 Critical Fixes

### 🔧 Notification System Rewrite (Goodbye `pkg_resources.DistributionNotFound`)

**The Bug** *(reported by justkub96, timun, unisexbrian, Detteride and others):*

```
pkg_resources.DistributionNotFound: The 'win10toast' distribution was not found
```

The tool would either fail at startup or sit idle forever. Root cause: `win10toast` (an unmaintained library) does a runtime `pkg_resources` lookup that PyInstaller-bundled executables don't reliably ship with on modern Windows 11. Even users who had `win10toast` installed in pip would hit this in the bundled `.exe`.

**The Fix:**

- Replaced `win10toast` with **`winotify`** — a pure-Python library with no `pkg_resources` runtime calls
- Added **triple-layered defense** so a notification failure can NEVER block grass generation again:
  1. **Import guard** — `except Exception` (not just `ImportError`) so distribution-not-found, missing-DLL, or any other init failure is caught
  2. **Instantiation probe** — at `Notifier.__init__` time, fail fast at startup instead of 90 minutes into a generation
  3. **Per-call guard** — every `toast.show()` is wrapped, errors logged at debug, generation proceeds
- **Auto-detect Wine/Proton** and disable toasts gracefully

**Impact:**

- Tool will start and run even when notifications are completely broken
- Steam Deck and Wine users now get a clean experience
- Bundled `.exe` is far more reliable on Windows 11

### 🔧 Fast Crash Detection (Saves Minutes Per Retry)

**The Bug** *(reported by justkub96):*

> "10 mins for freeze detection was wasting too much time so I reduced it to 5 mins which still wastes a lot of time. […] this tool doesn't detect quitting of the game immediately, instead it waits for freeze timeout which is 5 mins then reports there was a crash and restarts the game."

The tool only checked if Skyrim was alive **after** the full activity-inactivity timeout had elapsed. So when NGIO's own freeze detection killed the game at ~1.5 minutes, our tool wasted another 5+ minutes before noticing.

**The Fix:**

- Process liveness is now checked **every 5 seconds** during monitoring
- Confirmed only after **2 consecutive failed checks** (~10s) — absorbs psutil transient quirks (psutil can briefly report a process as dead during scheduling/paging)
- Re-checks the precache file before declaring a crash to handle the race where the plugin completes and the game exits in the same tick (we treat that as success, not crash — as we should)
- **Defensive wrapper**: if `is_process_running()` ever raises an unexpected exception, we treat the process as ALIVE (conservative — never false-positive a crash on a flaky read)
- **Power-user escape hatch**: set `NGIO_DISABLE_FAST_DEATH_DETECTION=1` to revert to the legacy "only check process death after activity timeout" behavior, in case any user hits an edge case

**Impact:**

- Crash detection: **~10 seconds** instead of 5+ minutes
- **Per-retry savings: 4–9 minutes**
- Overnight runs with 50-retry budgets save hours when crashes are common

### 🔧 Warmup Grace Period (Heavy Modlists Welcome)

**The Bug** *(also reported by justkub96):*

> "my game takes like 4 mins to lunch due to heavy modding and if game doesn't launch before the timeout, the tool restarts it before it can even begin generating. […] starting the timeout counter only after the first set of generation (if possible) would be really great."

The activity-inactivity timer started counting at function entry, so heavy modlists (4+ min boot) could blow the budget before generation even began.

**The Fix:**

- Activity timer now waits for the **first PrecacheGrass.txt write** before arming
- Hard cap of **10 minutes of warmup** so a truly hung launch still bails (instead of waiting forever)
- Clear log message: `✅ Generation started - first activity after 240s of boot`
- Process-death detection is **still active** during warmup, so a boot-time crash is caught immediately

**Impact:**

- Heavy modlists (250+ plugins) won't false-positive timeout
- The "no progress timeout" setting now means what it says — no progress *during generation*

## 🔬 Technical Notes

### What changed under the hood

| Component | Old | New |
|---|---|---|
| Notification backend | `win10toast` (unmaintained, `pkg_resources` runtime lookup) | `winotify` (pure-Python, PyInstaller-friendly) |
| Liveness check frequency | Every 5 minutes (full activity timeout) | Every 5 seconds, 2-check streak |
| Activity timer start | Function entry | First file write (with 10-min cap) |
| Notification failure handling | `ImportError`-only guard | Triple-layer defense, never raises |
| Wine/Proton support | Toasts crash | Toasts auto-disabled, sounds still work |

### New environment variables

| Variable | Purpose |
|---|---|
| `NGIO_DISABLE_FAST_DEATH_DETECTION=1` | Revert to legacy crash-detection (escape hatch) |

## 📋 Migration Notes

**For existing users:** No action required. Existing configs continue to work. The new LOD grass setting is opt-in and prompts you on first run.

**For users running from source:**

```bash
pip install -r requirements.txt
```

This will swap `win10toast` for `winotify`. You can also `pip uninstall win10toast` afterward to clean up.

**For users running the bundled `.exe`:** Just download and use. The bundle is self-contained.
